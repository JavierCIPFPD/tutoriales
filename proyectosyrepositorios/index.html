Control de versiones

¿Que es Git y para que sirve?

Git es un sistema de control de versiones distribuido. Guarda una fotografia de tu proyecto cada vez que tu lo decides, permitiendo deshacer errores, colaborar sin conflictos y mantener un historial legible de todo lo que ha cambiado.

Que hace Git

Registra cada cambio en tus archivos como un commit. Puedes ver quien cambio que, cuando y por que, y recuperar cualquier version anterior con un solo comando.

A diferencia de copiar carpetas a mano, Git guarda solo las diferencias entre versiones, lo que ahorra espacio y aclara el historial.

Las tres zonas de Git

Git separa el trabajo en tres areas que conviene entender antes de empezar:

  • Directorio de trabajo: donde ves y editas tus archivos.
  • Area de preparacion (stage): donde seleccionas que cambios guardar.
  • Repositorio local: donde quedan guardados los commits permanentemente.

Git vs GitHub

Git es la herramienta que se instala en tu ordenador. GitHub es un servicio en la nube donde puedes subir ese repositorio para compartirlo, colaborar o hacer una copia de seguridad.

Git puede usarse sin GitHub, pero GitHub necesita Git para funcionar.

Comandos basicos de Git

Estos son los comandos del dia a dia. Con ellos cubres el flujo completo de un proyecto sencillo.

INICIO

git init

Crea un repositorio Git vacio en la carpeta actual. Solo se ejecuta una vez al comenzar el proyecto.

git clone <url>

Descarga un repositorio remoto completo, con todo su historial.

ESTADO

git status

Muestra que archivos han cambiado, cuales estan preparados y cuales no.

git log --oneline

Lista el historial de commits de forma compacta, un commit por linea.

git diff

Muestra exactamente que lineas han cambiado en los archivos modificados.

CAMBIOS

git add .

Mueve todos los cambios al area de preparacion. Usa un nombre de archivo en lugar del punto para seleccionar solo uno.

git commit -m "mensaje"

Guarda los cambios preparados como un nuevo punto en el historial. El mensaje debe describir que cambio y por que.

git restore <archivo>

Descarta los cambios no guardados de un archivo y lo vuelve al estado del ultimo commit.

RAMAS

git branch

Lista todas las ramas del repositorio y marca la activa.

git checkout -b <nombre>

Crea una nueva rama y cambia a ella en un solo paso.

git merge <rama>

Integra los commits de otra rama en la rama activa.

REMOTO

git remote add origin <url>

Registra la URL del repositorio remoto con el nombre origin, el alias estandar para el servidor principal.

git push -u origin main

Sube los commits locales al servidor. El flag -u establece el seguimiento automatico para futuros push y pull.

git pull

Descarga los cambios del servidor y los integra automaticamente en la rama local.

CONFIGURACION

git config --global user.name "Tu Nombre"

Establece el nombre que aparecera en todos tus commits. Se guarda una sola vez en el sistema.

git config --global user.email "correo@ejemplo.com"

Registra el email vinculado a tus commits, que GitHub usa para asociarlos a tu cuenta.

Tutorial con VS Code

De cero a GitHub: paso a paso

Usamos el proyecto miprimerproyectoweb como referencia. Cada paso puede hacerse desde la interfaz grafica de VS Code o desde el terminal integrado.

01

Instalar Git en tu ordenador

Descarga Git desde git-scm.com e instalalo con las opciones por defecto. En Windows el instalador incluye Git Bash.

Para comprobar que la instalacion fue bien, abre el terminal y ejecuta:

git --version
02

Configurar tu identidad

Git asocia cada commit a un nombre y un email. Estos datos son solo informativos, no afectan al acceso. Ejecuta una sola vez: 

git config --global user.name "Ana Garcia"
git config --global user.email "ana@ejemplo.com"

El flag --global aplica esta configuracion a todos tus proyectos.

03

Abrir el proyecto en VS Code

En VS Code usa Archivo → Abrir carpeta… y selecciona miprimerproyectoweb. Asegurate de que la carpeta raiz del proyecto es la que abre VS Code, no una carpeta padre.

El icono de Control de codigo fuente (tercero en la barra lateral izquierda) mostrara No hay un repositorio Git hasta que lo inicializemos.

04

Inicializar el repositorio local

Tienes dos formas equivalentes de hacerlo:

Opcion A — Desde VS Code: en el panel de Control de codigo fuente, pulsa el boton Inicializar repositorio.

Opcion B — Desde el terminal:

git init

Git crea una carpeta oculta .git con toda la base de datos del repositorio. No la toques ni la borres.

05

Crear el archivo .gitignore

Antes del primer commit conviene indicarle a Git que archivos ignorar: carpetas de dependencias, archivos de configuracion local o secretos. Crea un archivo llamado .gitignore en la raiz con el contenido que necesites. Ejemplo minimo: 

node_modules/
.DS_Store
*.log

GitHub ofrece plantillas de .gitignore para cada lenguaje al crear un repositorio.

06

Hacer el primer commit

Ahora hay que llevar los archivos al area de preparacion y luego confirmarlos.

Opcion A — Desde VS Code:

  1. En el panel de Control de codigo fuente, verás todos los archivos bajo Cambios.
  2. Pulsa el icono + junto a Cambios para preparar todos.
  3. Escribe un mensaje en el cuadro de texto, por ejemplo: feat: estructura inicial del proyecto.
  4. Pulsa ✓ Confirmar (o Ctrl+Enter).

Opcion B — Desde el terminal:

git add .
git commit -m "feat: estructura inicial del proyecto"

Un buen mensaje de commit explica el que y el por que, no el como. La convencion Conventional Commits añade un prefijo como feat:, fix: o docs:.

07

Crear el repositorio en GitHub

  1. Accede a github.com e inicia sesion.
  2. Pulsa el boton verde New o el icono + en la esquina superior derecha.
  3. Escribe el nombre del repositorio: miprimerproyectoweb.
  4. Elige Public o Private segun tu preferencia (lo explicamos en la seccion siguiente).
  5. No marques ninguna opcion de inicializacion (README, .gitignore, licencia) ya que el proyecto ya existe en local.
  6. Pulsa Create repository.

GitHub mostrara una pantalla con instrucciones; copia la URL del repositorio.

08

Conectar el repositorio local con GitHub

Hay que decirle a Git donde esta el servidor remoto. En VS Code abre el terminal integrado con Ctrl + ` y ejecuta (sustituye la URL por la tuya): 

git remote add origin https://github.com/tuusuario/miprimerproyectoweb.git
git branch -M main

origin es el alias del remoto. El segundo comando renombra la rama principal a main, el nombre estandar en GitHub.

09

Subir el proyecto a GitHub (primer push)

Opcion A — Desde VS Code:

  1. En el panel de Control de codigo fuente, despliega el menu ···.
  2. Selecciona Push. VS Code pedira autenticacion si es la primera vez.

Opcion B — Desde el terminal:

git push -u origin main

El flag -u (upstream) guarda la configuracion para que los siguientes pushes solo necesiten git push.

Actualiza la pagina de GitHub: veras todos tus archivos ya publicados.

10

El flujo diario de trabajo

A partir de este punto, el ciclo habitual es sencillo:

✏️ Edita archivos en VS Code
→
➕ Stage los cambios (+ en el panel o git add)
→
💬 Escribe un mensaje y haz commit
→
☁️ Push para subir a GitHub

Si trabajas en varios dispositivos o en equipo, haz siempre git pull antes de empezar a editar para sincronizar los cambios del servidor.

Organizar en GitHub

Repositorios publicos y privados

GitHub ofrece dos tipos de visibilidad. La eleccion depende de a quien quieres dar acceso y de si deseas que el proyecto sea visible en tu perfil.

Publico Repositorio publico

Cualquier persona puede ver el codigo, clonar el repositorio y leer su historial. Solo tu (y los colaboradores que invites) podeis hacer push.

Usa un repositorio publico cuando:

  • Quieras mostrar el proyecto en tu perfil de GitHub como parte de tu portfolio.
  • Sea un proyecto de codigo abierto y quieras que otros contribuyan.
  • Sea un ejercicio, practica o proyecto educativo sin datos sensibles.
  • Quieras usar GitHub Pages de forma gratuita (requiere repositorio publico en la cuenta gratuita).

Privado Repositorio privado

Solo tu y las personas que invites expresamente pueden ver el repositorio. No aparece en tu perfil publico.

Usa un repositorio privado cuando:

  • El proyecto contenga datos sensibles o credenciales.
  • Sea un proyecto de cliente o trabajo con codigo propietario.
  • Este en fase inicial y no quieras publicarlo todavia.
  • Colabores con un equipo cerrado sin querer exposicion externa.

Consejo: organiza tus repositorios con claridad

  • Usa nombres descriptivos y en minusculas con guiones: mi-proyecto-web, no prj1_v2_final.
  • Añade siempre un README.md con una descripcion breve, instrucciones de instalacion y uso.
  • Añade una licencia en proyectos publicos (MIT es la mas permisiva y habitual).
  • Usa topics (pestana About del repositorio) para etiquetar el lenguaje y el area del proyecto.
  • Agrupa proyectos relacionados bajo una organizacion si trabajas con mas personas.

Publicacion gratuita

Publica tu proyecto en GitHub Pages

GitHub Pages sirve los archivos estaticos de tu repositorio como una web publica sin necesidad de pagar un hosting. Perfecto para proyectos HTML, CSS y JavaScript como miprimerproyectoweb.

01

Accede a Settings del repositorio

En GitHub, abre el repositorio miprimerproyectoweb y pulsa la pestana Settings (la ultima del menu superior, con icono de engranaje).

02

Abre la seccion Pages

En el menu lateral izquierdo de Settings, busca la seccion Code and automation y pulsa Pages.

03

Configura la fuente de publicacion

En Build and deployment → Source selecciona Deploy from a branch.

En Branch, elige main y en el selector de carpeta deja / (root).

Pulsa Save.

04

Espera el despliegue

GitHub tarda entre 30 segundos y 2 minutos en compilar y publicar la web. Recarga la pagina de Settings → Pages hasta que aparezca el mensaje "Your site is live at…" con la URL definitiva.

La URL tendra siempre la forma:

https://tuusuario.github.io/miprimerproyectoweb/
05

Actualizaciones automaticas

A partir de ahora, cada vez que hagas git push a la rama main, GitHub Pages volvera a desplegar automaticamente la web con los nuevos cambios.

No necesitas repetir la configuracion de Pages: el despliegue continuo ya esta activado.

06

Dominio personalizado (opcional)

Si tienes un dominio propio, puedes vincularlo en la misma seccion Pages añadiendo un archivo CNAME en la raiz del repositorio con el nombre de tu dominio y configurando los registros A o CNAME en tu proveedor de DNS.
